# AGENTS.md

Guidance for AI coding agents working in this repository.

## Project

İlk Onbir — a fantasy football lineup builder. Users pick a real club, see its
auto-assembled starting XI on a pitch, swap players in from the bench or from
other clubs, switch formations, and watch total squad market value update live.

- **Backend**: `backend/` — FastAPI + SQLAlchemy (async) + PostgreSQL/SQLite, Alembic migrations.
- **Frontend**: `frontend/` — React (Vite) + Tailwind v4 + Framer Motion.
- No auth: lineups are guest/session-based, the lineup id lives in the browser's
  `sessionStorage` keyed per club.

## Setup & run

```powershell
# Backend
cd backend
.\.venv\Scripts\pip install -r requirements.txt
.\.venv\Scripts\python -m app.seed              # demo data (10 hand-picked clubs)
.\.venv\Scripts\python -m uvicorn app.main:app --reload --port 8000

# Frontend
cd frontend
npm install
npm run dev        # http://localhost:5173
```

```powershell
# Optional: PostgreSQL + transfermarkt-api via Docker
docker compose up -d db transfermarkt-api
```

## Testing & linting

There is no test suite and no linter configured in either `backend/` or
`frontend/` (no pytest, no ruff/flake8, no ESLint). Don't invent commands for
these — verify changes by running the dev servers and exercising the
affected endpoint/page directly.

## Migrations

```powershell
cd backend
.\.venv\Scripts\alembic revision --autogenerate -m "message"
.\.venv\Scripts\alembic upgrade head
```
Review autogenerated migrations before applying: a new `NOT NULL` column on a
table that already has rows needs a `server_default` (add it, backfill, then
you can drop the default) — autogenerate does not add one for you.

## Data sync

Real data isn't pulled automatically; it only updates via explicit sync runs:
`POST /admin/sync` (runs in-process as a background task) or the standalone
scripts in `backend/` (`run_full_clubs_sync.py`, `run_foreign_clubs_sync.py`,
`run_fdo_sync.py`, `resync_club.py <team_id>`, `run_retry_unmatched_clubs.py`,
`cleanup_duplicate_clubs.py`). **Prefer the standalone scripts for anything
that runs more than a few seconds**: editing any backend `.py` file while a
sync is running inside the dev server triggers `uvicorn --reload` and silently
kills the in-flight background task.

## Architecture notes

### Two external data sources, kept deliberately separate

- **API-Football** (`app/services/api_football.py`): Süper Lig clubs/squads/coaches
  (`API_FOOTBALL_LEAGUES=203`) plus goal/assist leaderboards for Süper Lig + the 3
  UEFA cups (`API_FOOTBALL_STAT_LEAGUES`, kept as a separate league list from the
  club sync on purpose — UEFA cup IDs must never end up in the club-sync league
  list or they'd create phantom "clubs"). The free plan only allows **seasons
  2022–2024** for `/players/topscorers` and `/players/topassists` — but
  `/players/squads` and `/coachs` are *not* season-gated and always return the
  current squad/coach. Don't assume the season lock applies to every endpoint.
- **football-data.org** (`app/services/football_data_org.py`): the current
  2025-26 season where API-Football is stuck on 2024-25. `sync_foreign_clubs`
  syncs all clubs+squads for PL/La Liga/Bundesliga/Serie A/Ligue 1 — one HTTP
  request per league returns every team's full squad, much cheaper than
  API-Football's one-request-per-team model. `sync_football_data_org_scorers`
  covers those 5 leagues plus Champions League. Its response includes an
  `assists` field, but it's often `null` and sorted by goals, not assists — it
  is intentionally discarded (hardcoded to `0`) instead of shown as a real
  assist ranking. **Always pass an explicit `season=`** — without it the API
  picks "the current season" by calendar, which during the summer transfer
  window resolves to a season that hasn't started yet (zero data).

Both sources populate `Club`/`Player`, so each has its own external-id column
(`external_api_football_id`, `external_football_data_org_id`) rather than
sharing one. Follow this pattern for any third source instead of overloading
an existing id column. `PlayerSeasonStat` similarly has a `source` field baked
into its unique constraint, because the two sources use incompatible league-id
numbering for the same competition.

### Squad sync is diff-based

`sync_clubs_and_players` (API-Football) and `sync_foreign_clubs`
(football-data.org) fully replace a club's roster on every run — diff the
fresh squad against the DB, delete players no longer present (nulling any
`LineupSlot.player_id` pointing at them first). This is how transferred-out
players disappear. New sync code should follow the same pattern rather than
only inserting/updating.

### Club matching pitfall

`seed_data.py` hardcodes real-looking API-Football team IDs for its 8 demo
foreign clubs (Real Madrid, Barcelona, Arsenal, Man City, Liverpool, Bayern,
PSG, Inter) even though those clubs were never synced from API-Football. A
query filtering `WHERE external_api_football_id IS NULL` to find "unclaimed
seed clubs" wrongly skips these 8 and creates duplicates instead of updating
them — filter on the external-id column of the source you're matching against.

### Auto-lineup assignment (`app/services/lineup_service.py`)

`build_default_assignment` matches each player's `detail_position` (a Turkish
label, e.g. "Stoper", "Sol Kanat", "Merkez Orta Saha") against the formation
slot's `label`, in tiers (exact → same side → neutral → wrong side), tie-broken
by `market_value` descending. Any new position-mapping table (see
`_POSITION_MAP` in `football_data_org.py` and `_DETAIL_MAP`/`_POSITION_MAP` in
`api_football.py`) must emit these same Turkish labels or matching silently
degrades. `remap_assignment` handles formation switches: same slot key first,
then role-compatible leftovers, then anything left.

### `sync_market_values` session-expiry gotcha

Fetches club ids up front, then re-fetches each club fresh with `session.get()`
inside the loop. Don't load the full `Club` list once and iterate those ORM
objects across multiple commits — one club's httpx error triggers
`session.rollback()`, which expires every object loaded in that session, and
touching a different previously-loaded object afterward raises
`MissingGreenlet` (invisible at small scale, broke at 96+ clubs).
`_find_club_id` also retries with a simplified name (stopwords/founding-year
stripped) when the literal name returns nothing, since many football-data.org
official names ("FC Internazionale Milano") don't match transfermarkt's search
index at all.

### Formation/pitch coordinates

`Formation.position_slots` is a JSON list of `{key, label, role, x, y}`, x/y as
percentages for absolute positioning on the pitch SVG. `formations_data.py` is
the source of truth for the 5 built-in formations — adding one there is
enough, the frontend renders slots generically from this data.

### Frontend routing

`App.jsx` keys `<Routes>` on `location.pathname`, so every navigation fully
remounts the page tree (enables Framer Motion page-transition animations) —
component state does not survive a route change. The lineup page
(`/club/:clubId`) hides the top nav header for a full-screen pitch view.
